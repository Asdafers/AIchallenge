## Findings

### Finding 1 - BLOCKER - MCP compatibility within Agent Runtime
The ADK `root_agent` relies on an MCP stdio server wrapper. Agent Runtime's ability to host and execute this subprocess-based MCP wrapper is unknown and highly risky.
*   **Mitigation:** The immediate feasibility spike MUST attempt to execute an MCP tool call within the deployed Agent Runtime environment. If it fails due to container sandbox restrictions, document the exact error and immediately pivot to the Cloud Run fallback.

### Finding 2 - MAJOR - ADK Evaluation is not explicitly prioritized in the spike
The product thesis states: "No useful insights without good data". To prove this, the deployment must demonstrate measurable data quality improvement. While the demo UI is planned to show this, a formal ADK evaluation artifact is stronger proof for the judges.
*   **Mitigation:** Add an ADK Evaluation run to the immediate spike or early deployment phases to generate a concrete evaluation artifact comparing the agent's output against a static baseline.

### Finding 3 - MAJOR - Custom `BaseAgent` serialization risks
`methodic/agent.py` contains multiple custom `BaseAgent` implementations (`ExtractorStep`, `TurnCheckerStep`, etc.). Agent Runtime may have specific requirements or limitations regarding the deployment and serialization of custom Python classes in the agent graph.
*   **Mitigation:** The Agent Runtime spike must execute a full pass of the `fieldwork_loop` to ensure all custom step logic is successfully executed in the cloud environment.

### Finding 4 - NOTE - Agents CLI Scaffold Porting
Porting Agents CLI conventions into an existing repository can be messy and distract from core goals.
*   **Mitigation:** Only port IAM roles and deployment configuration scripts. Do not alter the existing `methodic/` directory structure unless strictly required by Agent Runtime.

## Falsifiable Assumptions

1.  Agent Runtime supports the execution of local stdio-based MCP servers within its managed container environment.
2.  The existing `methodic/agent.py` graph, including all custom `BaseAgent` subclasses, can be deployed to Agent Runtime without significant refactoring.
3.  Agent Platform sessions can fully replace the exploratory `InMemorySessionService` without losing required context during the `fieldwork_loop`.
